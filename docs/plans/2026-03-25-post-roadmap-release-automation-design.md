# Post-roadmap — Release automation

## Goal

Add the first release automation batch as a manual GitHub Actions workflow that verifies the repo, synchronizes workspace versions, creates a release commit and tag, and opens a draft GitHub Release.

## Scope

- add a dedicated `workflow_dispatch` release workflow in GitHub Actions
- accept one explicit semver input:
  - `version`
- rerun the existing verification gates before any release side effects:
  - `pnpm lint`
  - `pnpm typecheck`
  - `pnpm --filter @dockeradmin/shared test`
  - `pnpm --filter @dockeradmin/api test`
  - `pnpm --filter @dockeradmin/web test`
  - `pnpm build`
- synchronize `version` in:
  - root `package.json`
  - `apps/api/package.json`
  - `apps/web/package.json`
  - `packages/shared/package.json`
- create one release commit:
  - `chore(release): vX.Y.Z`
- create one git tag:
  - `vX.Y.Z`
- create one draft GitHub Release with autogenerated notes

## Out of scope

- no npm publish
- no Docker registry publish
- no deploy automation
- no automatic changelog file generation
- no semantic-release or conventional-commit version inference
- no prerelease channel handling in this batch

## Design

### Release entrypoint

- add `.github/workflows/release.yml`
- trigger:
  - `workflow_dispatch`
- input:
  - `version`
- the version must be an explicit strict semver string such as `0.2.0`
- the workflow is allowed to release only from the repository default branch

### Workflow safety

- set:
  - `permissions: contents: write`
- set:
  - `concurrency: release`
- checkout with:
  - `fetch-depth: 0`
- validate before any mutation:
  - requested `version` is strict semver
  - requested `version` differs from the current repo version
  - tag `vX.Y.Z` does not already exist
- if verification fails at any point before git push, the workflow must stop without creating a release commit or tag

### Version synchronization

- add a repo-local Node script:
  - `scripts/release-version.mjs`
- the script owns:
  - strict semver validation
  - current-version comparison
  - synchronized JSON updates across the four `package.json` files
- keep file formatting stable:
  - two-space indentation
  - trailing newline

### Release side effects

- after verification and version synchronization:
  - configure git author as `github-actions[bot]`
  - commit the version bump
  - create tag `vX.Y.Z`
  - push commit and tag to the default branch
- then create a draft GitHub Release for the same tag with generated notes
- use the default `GITHUB_TOKEN`; no extra secret management is introduced in this batch

### Failure semantics

- no partial release side effects before verification passes
- a failure after push but before GitHub Release creation may leave:
  - the release commit
  - the release tag
  on the default branch without the draft release object
- this is acceptable for the first batch as long as the workflow fails loudly

## Testing

- add failing tests first for:
  - workflow existence and `workflow_dispatch` input shape
  - workflow permissions, concurrency, default-branch guard, and verification steps
  - workflow release steps:
    - version bump script
    - release commit
    - git tag
    - draft GitHub Release creation
  - version script semver validation
  - version script synchronized updates across the four package manifests

## Verification

- `node --test scripts/ci-workflow.test.mjs scripts/release-workflow.test.mjs scripts/release-version.test.mjs`
- `pnpm lint`
- `pnpm typecheck`
- `pnpm build`
- `pnpm docker:platform:up`
- `curl -fsS http://localhost/api/health`
- `curl -fsS http://localhost`
